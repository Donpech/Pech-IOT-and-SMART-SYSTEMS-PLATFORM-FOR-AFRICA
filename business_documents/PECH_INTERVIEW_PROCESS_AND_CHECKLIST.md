## SECTION 3: SCORING METHODOLOGY

### 3.1 Score Scale Definitions

| Score | Label | Definition | Indicator |
|---|---|---|---|
| **1** | Unacceptable | Does not meet minimum requirements. Fundamental gaps in knowledge or skills. | Cannot perform basic role functions |
| **2** | Below Expectations | Meets some requirements but has significant gaps. Would need extensive training. | Struggles with standard tasks |
| **3** | Meets Expectations | Competent. Can perform the role with normal onboarding and support. | Handles standard tasks independently |
| **4** | Exceeds Expectations | Strong candidate. Would add value quickly with minimal ramp-up. | Handles complex tasks, shows initiative |
| **5** | Exceptional | Outstanding. Top-tier talent who would elevate the entire team. | Demonstrates mastery, leadership, innovation |

### 3.2 Scoring Dimensions & Weights

| Dimension | Weight | What It Measures |
|---|---|---|
| **Technical Competence** | 40% | Role-specific skills, knowledge depth, problem-solving ability |
| **Behavioral & Soft Skills** | 25% | Communication, teamwork, adaptability, work ethic |
| **Cultural Alignment** | 20% | Values fit (Reliability, Quality, Africa-first, Ownership) |
| **Communication** | 15% | Clarity of expression, listening, ability to explain complex topics |

### 3.3 Minimum Passing Thresholds

| Role Level | Minimum Weighted Average | Meaning |
|---|---|---|
| **C-Level / Senior** | 3.5 | Must exceed expectations in most dimensions |
| **Mid-Level** | 3.0 | Must meet expectations across the board |
| **Junior** | 2.5 | Must show competence and strong learning potential |
| **Intern** | 2.0 | Potential-weighted — attitude and learning speed matter most |
| **Field Agent** | 2.5 | Must demonstrate reliability, communication, and local knowledge |

### 3.4 Weighted Score Calculation

> **Formula:** Weighted Score = (Technical × 0.40) + (Behavioral × 0.25) + (Cultural × 0.20) + (Communication × 0.15)
>
> **Example:** A mid-level candidate scores: Technical=4, Behavioral=3, Cultural=4, Communication=3
> Weighted Score = (4×0.40) + (3×0.25) + (4×0.20) + (3×0.15) = 1.60 + 0.75 + 0.80 + 0.45 = **3.60** (PASS — above 3.0 threshold)

### 5.2 Take-Home Assignment Guidelines

> **Instructions to share with candidates:**
>
> 1. You have **48 hours** from receipt to submit your solution
> 2. Submit via a **private GitHub repository** (add our reviewer as collaborator) or email
> 3. Include a **README** explaining your approach, trade-offs, and what you'd improve with more time
> 4. Use **any resources** you want (Google, docs, StackOverflow) — but the code must be YOUR work
> 5. We value **clean, readable code** over cleverness. Show us how you'd write production code.
> 6. A **live review session** will follow where you walk us through your solution and make modifications
>
> **Anti-Plagiarism Notice:** During the live review, you will be asked to explain your code line-by-line and make real-time modifications. If you cannot explain or modify your own submission, this will be flagged as a disqualifying concern.

### 7.2 Technical Fraud Detection Methods

| # | Method | When to Use | Red Flag Indicators |
|---|---|---|---|
| 1 | **Line-by-line code walkthrough** | All technical take-home reviews | Cannot explain own code, long pauses, vague answers |
| 2 | **Live modification request** | All technical reviews | Cannot make simple changes to their own code |
| 3 | **"Why" chain questioning** | All rounds | Ask "why?" 3 levels deep on any claim — fraudsters collapse at level 2–3 |
| 4 | **Pair programming** | Senior/Mid engineering | Observe their real-time thinking process |
| 5 | **Past project deep-dive** | All technical | Ask about edge cases, bugs encountered, architecture decisions |
| 6 | **GitHub activity audit** | Engineering roles | Check: commit frequency, code style consistency, contribution history |
| 7 | **Portfolio recreation** | UI/UX designers | Ask to recreate a small element from their portfolio live in Figma |
| 8 | **Hardware hands-on** | IoT/Embedded roles | Give them an actual ESP32 board and ask basic operations |
| 9 | **Plagiarism check** | Take-home assignments | Run code through plagiarism detectors, check for cookie-cutter patterns |
| 10 | **AI-generated content check** | Written assessments | Look for ChatGPT-style phrasing, overly polished but shallow analysis |

### 7.3 Red Flag Severity Matrix

| Severity | Indicators | Action |
|---|---|---|
| **Critical (Immediate Reject)** | Cannot explain own code at all; caught with someone else's GitHub; identity mismatch | Terminate interview, document, blacklist |
| **High (Likely Reject)** | Significant CV discrepancies confirmed; unable to make basic live modifications; claims skills they demonstrably lack | Complete interview for documentation, likely reject |
| **Medium (Investigate Further)** | Minor date inconsistencies; nervous during live coding (could be anxiety); unverifiable employer | Proceed but add extra verification steps |
| **Low (Monitor)** | Slightly embellished responsibilities; minor gaps in knowledge; cultural red flags | Note for probation monitoring if hired |
